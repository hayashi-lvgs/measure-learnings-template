# PDFからカードを作る（Claude Codeに読ませて綺麗なカードにする）

## なぜこの方法なのか

サイト（ブラウザ）のドロップ取り込みは、AIではなく**正規表現の仕分け機**です。意味を理解しないので、施策前後が混ざる・URLが見切れる・概要が抜ける、といった荒い下書きにしかなりません。しかも静的サイトの制約（`file://`・`fetch()`禁止・APIキーを埋め込めない）で、**その場でAIを呼ぶことはできません**。

「内容を理解した綺麗なカード」を作れるのは **Claude Code側（本物のモデル）** だけです。Claude CodeはPDFを直接読めるので、**ブラウザのドロップを使わず**、この手順でカードを作るのが最良です。

## 使い方

1. PDF（検証用シート等）をこのリポジトリのどこか（例 `pdf/` フォルダ）に置く
2. このリポジトリで Claude Code を開く
3. 下のプロンプトを貼り、PDFのパスを指定する

## コピペ用プロンプト

```
docs/施策データの書き方.md を読んでから、次のPDFを施策カードにしてください: pdf/<ファイル名>.pdf

【重要】いきなりカードを書かないこと。次の順で進めてください。

■ ステップ1: まずPDFを最後まで読み、「理解サマリ」を日本語で返す（カードはまだ書かない）
  - この施策は何をしたのか（1〜2行）
  - 施策前の状態 → 施策後の状態（必ず分けて書く）
  - 結果（数値・有意差）と、勝ち/引き分け/負けの判断根拠
  - PDFに"事実として書かれていること" と "考察・推測" を分けて列挙
  - 読み取れなかった／曖昧な項目があれば明示する
  ここで一度止まって、私の確認を待つ。

■ ステップ2: 私がOKしたら、data/measures.js の MEASURES 配列末尾にカードを1件追加する
  必須項目チェック（1つでも埋められなければ、勝手に創作せず私に質問する）:
  - [ ] title（何の施策か一目でわかる）
  - [ ] background（📝施策概要：前提と課題。ここが空だとカードが理解不能になる）
  - [ ] hypothesis（仮説）
  - [ ] result（win/draw/loss）と adoptionLabel（採用判断。PDFに無ければ「要確認」と書く）
  - [ ] factuals（📋事実のみ。PDFに書かれていないことは入れない）
  - [ ] metrics（主要数値）
  - [ ] rootCause（なぜ勝った/負けたか）

  ルール（厳守）:
  - 事実(factuals) と AI推論(hypothesisInsights) を絶対に混ぜない。推測は hypothesisInsights に入れる
  - 施策前と施策後を1つの文に混ぜない
  - URL（Asana/Figma/Optimizely等）は factuals に入れず links 配列に入れる
  - result は PDF に採用判断が明記されている場合のみ確定。無ければ draw + adoptionLabel を「要確認」
  - designInsights は、うちの事業部のユーザー前提で ✅有効/⚠️注意/❌NG を書く（機械語尾判定のコピペ禁止）
  - 新しい metrics キーを使うなら assets/js/app.js の METRIC_LABELS に日本語ラベルを追記
  - fetch()/type="module" は使わない。データ構造は docs/施策データの書き方.md に従う

■ ステップ3: index.html をブラウザで開いてカードが正しく出るか、私が確認する手順を教える
```

## ポイント

- **ステップ1で必ず止める**のが肝。カードを書く前に「Claudeがちゃんと理解できているか」を人が確認できる。ここで誤読を潰せば、後段のカードは綺麗になる。
- 必須項目が埋まらないときに**Claudeが創作で埋めない**よう、質問させる。事実と推論の境界を守るため。
- スクショを付けたい場合は、施策前後の比較画像を `images/<ID>-before-after.png` に置き、`data/measures.js` の `SCREENSHOTS` に1行追加するよう依頼する。

## 補足：ブラウザのドロップ取り込みはいつ使う？

Claudeを使わずにサッと下書きだけ見たいときの**クイックプレビュー**用。そのブラウザにしか残らず、内容も荒いので、共有する正式カードは必ず上記のClaude手順で仕上げること。
