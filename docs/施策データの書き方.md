# 施策データの書き方（data/measures.js）

1施策 = `MEASURES` 配列のオブジェクト1つ。`tools/import.html` が生成する下書きも同じ形。

## 3原則（このサイトの核・崩さない）

1. **事実(📋)とAI推論・仮説(🤔)を明示的に区別する** — PDF等に書かれていない解釈を `factuals` に入れない。
2. 各施策に**デザイン観点の活かし**(✅有効/⚠️注意/❌NG)を、自事業部のユーザー前提で付ける。
3. **横断で学びをパターン化**する（`INSIGHTS`）。

## 施策オブジェクトのフィールド

```js
{
  id: "ID-001",                    // 施策の一意なID（事業部の管理番号など）
  title: "施策名",
  type: "施策タイプ",               // 例: "CTA削減" "バナー挿入" … サイドバーのフィルタになる
  pages: ["対象ページ"],
  devices: ["SP", "PC"],           // "SP" / "PC"
  result: "win",                   // win=勝ち / draw=引き分け / loss=負け
  adoptionLabel: "本番採用 など採用判断",
  period: "2025/01/01〜01/14（14日間）",
  hypothesis: "立てた仮説",
  hypothesisIsAI: false,           // 仮説がAI推論なら true
  background: "施策背景",
  factuals: [                      // 📋 事実だけ（PDF/データに書かれていること）
    "有意差なし（p=0.12）",
    "CVR 0.035% → 0.064%"
  ],
  hypothesisInsights: [            // 🤔 AI推論・仮説（事実ではない解釈）
    "…だった可能性がある"
  ],
  rootCause: "なぜ勝った/負けたかの要因（言語化）",
  metrics: {                       // カードのメトリクス表示。キーの日本語名は app.js の METRIC_LABELS
    cvrImprove: "+182%",
    significance: "有意差なし"
  },
  links: [                         // 🔗 関連リンク（Asana/Figma/Optimizely等）。factualsに混ぜない
    "https://app.asana.com/..."
  ],
  nextActions: ["次にやること"],
  designInsights: [                // デザイン観点。type: good(✅) / caution(⚠️) / bad(❌)
    { type: 'good', text: '…' },
    { type: 'bad',  text: '…' }
  ]
}
```

- 新しい指標キー（`metrics` のキー）を使ったら、`assets/js/app.js` の `METRIC_LABELS` に日本語ラベルを追記する。
- スクショを付けるなら `images/` に画像を置き、`SCREENSHOTS` に `"ID-001": "images/ID-001-before-after.png"` を1行足す。

## PDF取り込みの下書きを仕上げる

`tools/import.html` が出す下書きには `autoExtracted: true` が付く（機械抽出・未レビューの目印）。
判定・事実/推論の仕分け・デザイン観点を人＋Claudeで仕上げ、`🔧候補:` の接頭辞と `autoExtracted` 行を削除する。
